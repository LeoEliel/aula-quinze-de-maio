Teste de Software

Da auto-avaliação à instrumentação automatizada
No Sprint 7 vocês fizeram assessment de maturidade contra TMMi e DORA 2024, e
o exercício expôs lacunas concretas e mensuráveis nas áreas de processo do TMMi Level 2
— especialmente Test Policy and Strategy, Test Design and Execution documentadas e,
sobretudo, ausência de rastreabilidade formal entre requisito, caso de teste e evidência. O
Sprint 8 ataca essas lacunas exatas, mas não como exercício documental no Word — como
instrumentação automatizada no repositório de código. A escalada é deliberada: deixa-se de
descrever maturidade e passa-se a implementá-la com ferramentas de mercado.

Quatro camadas instrumentais articuladas
Camada 1 — Especificação formal pelo padrão internacional vigente: ISO/IEC/IEEE
29119-3:2021, que sucedeu o IEEE 829-2008 como referência mundial para documentação
de teste de software (ISO, 2021).
Camada 2 — Testes de contrato com Schemathesis sobre OpenAPI 3.1: a evolução
property-based do Sprint 3 agora aplicada ao nível HTTP, gerando milhares de requisições
automaticamente a partir do schema da API (DYGALO, 2024).
Camada 3 — Matriz de rastreabilidade requisito ↔ caso ↔ evidência automatizada
via GitHub Actions: o CI deixa de ser apenas executor de testes e passa a ser orquestrador de
evidência de auditoria, com o pipeline gerando a matriz a cada commit.
Camada 4 — Position paper técnico em diálogo com literatura contemporânea sobre
continuous testing maturity, com leitura obrigatória de Nayak et al. (2024) publicado pelo
IEEE em 2024 sobre sustentabilidade de pipelines de teste contínuo.

Como ler este roteiro
Este roteiro tem uma lógica de progressão. Cada bloco prepara o seguinte. Você vai
encontrar blocos de cinco naturezas, e precisa tratá-los de formas diferentes:
Blocos azuis trazem conceito ou continuidade. Leia com atenção; não há resposta a
registrar.
Blocos rosa trazem investigação técnica do aluno. Você precisa registrar sua resposta
no arquivo Word de entrega antes de avançar. A resposta é avaliada por consistência com os
autores e padrões indicados (ISO, 2021; DYGALO, 2024; NAYAK et al., 2024; OpenAPI
Initiative; GitHub Docs) e pela qualidade do raciocínio.
Blocos amarelos trazem atenção técnica ou troubleshooting.
Blocos roxos trazem comandos, prompts ou esqueletos de configuração para adaptar.
NUNCA cole o conteúdo bruto — você precisa preencher com os dados do seu repositório.
Blocos verdes trazem resultado esperado ou reflexão final. Eles são especialmente
importantes — é como você sabe que está no caminho certo sem o professor ao lado.
Atenção: este sprint exige conta gratuita no GitHub ou similiar. As fontes primárias
estão em acesso aberto (URLs nas Referências). Respostas sem ancoragem nos padrões e
autores indicados, sem evidência empírica de pipeline rodando no seu repositório, ou
construídas como paráfrase superficial dos textos lidos não serão consideradas tecnicamente
válidas.

Camada 1 — Conceito: ISO/IEC/IEEE 29119-3:2021 — Test Documentation
A ISO/IEC/IEEE 29119-3:2021 é a norma internacional vigente para documentação
de teste de software. Sucedeu formalmente o IEEE Standard 829 (1998 e 2008) e foi publicada
conjuntamente pela ISO, pela IEC e pelo IEEE — três das maiores organizações de

A norma estabelece templates para os seguintes documentos de teste, articulados em
três níveis (organizational, project, dynamic):
Test Policy: intenção estratégica da organização quanto à qualidade dos testes.
Test Plan: escopo, abordagem, recursos e cronograma da campanha de teste do
projeto.
Test Design Specification: técnicas, condições de teste e cobertura projetada.

Test Case Specification: identificador, descrição, pré-condições, dados de entrada,

Camada 2 — Conceito: Schemathesis e contract testing sobre OpenAPI
Schemathesis é a ferramenta open-source que aplica testes baseados em propriedade
automaticamente a APIs HTTP a partir do seu schema OpenAPI (DYGALO, 2024).
Documentação oficial em https://schemathesis.readthedocs.io/.
É a evolução direta do que vocês fizeram com Hypothesis no Sprint 3: lá, vocês
formularam invariantes manualmente e o Hypothesis gerava valores. Aqui, vocês descrevem
a

API

uma

única

vez

em

OpenAPI

3.1

(especificação

em

https://spec.openapis.org/oas/v3.1.0) e o Schemathesis gera milhares de requisições HTTP
automaticamente, testando se a API respeita o próprio contrato declarado — uma forma
rigorosa de teste de aceitação contratual no sentido do BSTQB (2023, Cap. 2).
Para este sprint, vocês escreverão o openapi.yaml manualmente, observando o
comportamento atual da API e traduzindo-o em contrato declarativo formal. Esse exercício é
o coração da Camada 2 — especificação de contrato observado é, ela mesma, um artefato de
teste, contraparte HTTP do que a Camada 1 faz com casos de teste pelo padrão 29119-3. O
ato de escrever o YAML obriga a declarar explicitamente cada endpoint, cada parâmetro,
cada código de resposta e cada formato de payload; essa declaração explícita é o oráculo que
o Schemathesis usará para gerar centenas de requisições e detectar divergências entre contrato
e comportamento real.

Camada 3 — Conceito: matriz de rastreabilidade automatizada no CI
Matriz de rastreabilidade requisito ↔ caso de teste ↔ evidência é um instrumento
clássico de auditoria, exigido por TMMi Level 3 e por normas regulatórias bancárias.
Historicamente, era mantida em planilhas Excel atualizadas manualmente — fonte
permanente de inconsistência entre o que estava no documento e o que estava no código.
A escalada moderna é gerar a matriz automaticamente no pipeline CI, a cada commit,
lendo (i) os IDs de teste especificados no caderno 29119-3, (ii) os resultados de execução do

pytest agregado e (iii) os requisitos rastreados via marcadores (pytest markers ou docstrings
padronizadas). GitHub Actions é a plataforma CI mais usada do mundo open-source em
2024-2025 (documentação oficial em https://docs.github.com/en/actions).
A virada conceitual é importante: o CI deixa de ser apenas executor da suíte e passa a
ser produtor de evidência de auditoria. A maturidade do processo é demonstrada a cada
commit, não declarada em reunião.

Atividade prática — esqueleto do pipeline GitHub Actions
Crie um repositório novo no GitHub chamado internet_banking_tests (público ou
privado), faça commit do conteúdo da sua pasta local. Dentro da pasta, crie o subdiretório
.github/workflows/ e nele um arquivo test.yml com o seguinte esqueleto (adapte ao seu
Python, ao seu requirements.txt e ao seu openapi.yaml):
name: Testes do internet banking
on: [push, pull_request]
jobs:
test:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- uses: actions/setup-python@v5
with: { python-version: '3.11' }
- run: pip install -r requirements.txt
- run: pip install schemathesis
- run: python app.py &
- run: sleep 3
- run: pytest -v --junit-xml=report.xml
- run: schemathesis run openapi.yaml --base-url=http://127.0.0.1:5000 --checks
all
- run: python scripts/gera_matriz.py > matriz_rastreabilidade.md
- uses: actions/upload-artifact@v4
with: { name: evidencia, path: 'report.xml matriz_rastreabilidade.md' }

Atenção — armadilhas comuns na configuração do CI
Primeiro: o servidor da API precisa rodar em background no runner (uso de "&" no
comando) e dar tempo de inicialização (sleep 3) antes de Schemathesis tentar conectar — sem
isso, o pipeline falha por connection refused. Segundo: o requirements.txt precisa listar todas
as dependências do app e dos testes (incluindo as instaladas ao longo dos Sprints 1 a 5 —
pytest, hypothesis, requests, pytest-bdd — mais schemathesis deste sprint) — se faltar uma,
o runner falha em pip install. Terceiro: actions/upload-artifact espera caminhos separados por
espaço em string única, não como lista YAML — formate como mostrado. Quarto: se o
repositório for privado, garanta que a aba Actions está habilitada nas Settings.
