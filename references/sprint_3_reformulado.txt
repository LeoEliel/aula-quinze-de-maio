Roteiro Sprint 2 - Property-based testing com Hypothesis

Objetivo: até aqui o GitHub Copilot gerou testes baseados em exemplos a partir
da sua Q8 (Sprint 1) e a fixture de isolamento a partir da sua observação crítica (Sprint 2).
Neste sprint, a ferramenta vai gerar testes baseados em propriedades a partir de invariantes
do domínio bancário que você mesmo formulará em linguagem natural. O salto é
epistemológico: deixar de pensar em casos pontuais e passar a pensar em propriedades
universais que o sistema precisa respeitar para qualquer entrada possível, sustentando a
análise em MacIver et al. (2019).

Conceito essencial: testes baseados em exemplos verificam casos específicos fixos
- “transferir 100 com saldo 1000 deve resultar em saldo 900”. Testes baseados em
propriedades verificam invariantes - “para qualquer valor positivo menor ou igual ao
saldo, a transferência deve ser aceita e a soma total dos saldos deve ser conservada”. A
biblioteca Hypothesis gera centenas de inputs automaticamente e, ao encontrar um
contraexemplo que viola a propriedade, faz shrinking - reduz o contraexemplo ao menor
caso possível que ainda quebra a propriedade, facilitando o diagnóstico (MacIver et al.,
2019, p. 2).

Invariante 1 - Rejeição de valor não-positivo no /transferencia
Para qualquer valor não-positivo (zero ou negativo) submetido ao endpoint /transferencia,
o sistema deve sempre retornar HTTP 422 — valor inválido.

Invariante 2 - Resposta universal a conta inexistente no /saldo
Raciocínio: no Sprint 1 a IA gerou um único teste para conta inexistente - algo
como test_saldo_conta_inexistente usando um único ID específico (por exemplo, 99). Isso
é um exemplo pontual: comprova que para aquele ID o retorno é 404. Mas não comprova
que todo ID fora do conjunto cadastrado retorna 404 - o sistema poderia, em tese, ter um
caminho de código defeituoso para um ID específico (ex.: ID 7, ID 100, ID 9999) e a suíte
da IA nunca descobriria. A invariante universal afirma que o comportamento se repete
para qualquer ID inteiro fora dos IDs cadastrados {1, 2, 3}.

Para qualquer ID inteiro que não pertença ao conjunto de contas cadastradas no
sistema (ou seja, ID ∉ {1, 2, 3}), o endpoint /saldo/<conta_id> deve sempre retornar
HTTP 404 - conta não encontrada.
Invariante 3 - Conservação de valor em transferência aceita
Raciocínio: uma transferência bancária é, matematicamente, uma redistribuição: o
que sai de uma conta entra na outra. Nada se cria, nada se perde. Se Alice tinha R$ 1000
e Bob tinha R$ 500, a soma é R$ 1500. Após uma transferência de R$ 300 entre elas
(aceita pelo sistema), a soma deve continuar sendo exatamente R$ 1500 - agora distribuída
como R$ 700 e R$ 800. Esse é o princípio de conservação de valor - uma das invariantes
mais críticas em qualquer sistema financeiro, porque sua violação significaria criação ou
destruição de dinheiro por defeito de software. A IA do Sprint 1 testou casos pontuais de
transferência com valores específicos, mas não verificou esta invariante universal - para
qualquer valor válido, ela deve se manter.

Para qualquer transferência aceita pelo endpoint /transferencia (resposta HTTP
200), a soma dos saldos das contas de origem e destino imediatamente após a operação
deve ser exatamente igual à soma dos saldos imediatamente antes da operação —
saldo(origem, antes) + saldo(destino, antes) = saldo(origem, depois) + saldo(destino,
depois).
